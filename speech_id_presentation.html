Why this problem matters

  • Speaker identification = which enrolled speaker produced a given utterance.
  • Core to meeting transcription, voice access control, and personalised voice assistants.
  • In a closed set, identification reduces to a search against enrolled voice data.
  • At larger scales, the naive exhaustive scan might become a bottleneck.
The two questions

1. Speed VS Accuracy: Can partitioning the database cut search cost without losing identification accuracy?

2. How much does the representation matter? Does the choice of different voice representations dominate the choice of search strategy?

Dataset & train / test split

  • VoxCeleb1: 1,251 celebrities' (690 M / 561 F) YouTube interviews, 16 kHz wav.
  • Leave-one-video-out split: the lexicographically last video directory of each speaker moves to the test set for speakers with more than one video.
  • Minimises leakage as much as possible: test utterances come from a recording session unseen during enrollment.

Three voice embedding models

192-d · task-specific

ECAPA-TDNN

SpeechBrain spkrec-ecapa-voxceleb. Conv + Res2Net blocks. Fine-tuned on VoxCeleb2 with additive angular-margin loss. L2-normalised.

512-d · task-specific

WavLM X-Vector

microsoft/wavlm-base-plus-sv transformer with an x-vector head (stats pooling + projection), fine-tuned on VoxCeleb1 for verification.

768-d · no fine-tuning

WavLM mean-pooled

Generic microsoft/wavlm-base-plus. Mean-pool all 12 layers, ≤20 s. Tests whether generic reps carry speaker identity.

Each model is stored in its own isolated PostgreSQL schema with a model-specific vector dimension.

Three search architectures

A

Brute-force

Full sequential scan over all 144,840 vectors. Cosine <=>, ORDER BY, LIMIT.

→
B

Metadata hierarchy

Pick nearest gender centroid, then top-P nationalities then query only that filtered candidate pool.

→
C

K-means routing

Find nearest cluster centroid (C = 1) and scan only that cluster. ~600 of 144,840 candidates.

Fair comparison: index scans are disabled before every run (enable_indexscan/indexonlyscan/bitmapscan = off) so PostgreSQL performs exact sequential scans regardless of architecture.

Aggregation modes & evaluation

How a winning speaker is chosen

Speaker centroid

Pre-computed per-speaker mean vectors held in memory.

Closest utterance

Return the speaker of the single nearest database result.

Vote

Top-25 results vote by speaker_id. Most votes wins, ties broken by best distance.

Metrics:

  • Top-1 % — correct speaker ranked first.
  • Hit@10 % — correct speaker in the top 10.
  • Macro-F1 % — per-speaker precision/recall.
  • Latency (ms/query) — perf_counter over all 8,676 queries.

Key analysis

Search architecture

Brute-force is near-perfect for ECAPA. K-means centroid nearly matches it at better speed. Metadata filtering collapses to ~33% — a routing error excludes the true speaker irreversibly.

Aggregation mode

Centroid is closest in accuracy but much faster (0.030 vs 37.81 ms). Vote hurts under K-means routing — dominant high-utterance speakers bias the majority.

Embedding quality

The representation constraint. WavLM mean-pooled caps at 29.66% Top-1 even with brute-force — no search strategy recovers from a weak representation.

Three-tier hierarchy: ECAPA-TDNN (best accuracy + lowest centroid latency) > WavLM X-Vector (97.42% Top-1, larger 512-d vectors → higher closest-search latency) > WavLM mean-pooled (unsuitable without fine-tuning).

Conclusions

  • The use of task-specific model embeddings is important.
  • Brute-force centroid is the practical optimum for ECAPA for a database of this scale.
  • K-Means centroid is the scalable real-time option.
  • Metadata hierarchy is not viable as a hard filter.
  • Vote aggregation degrades under K-Means routing due to dominant speaker.
Future work

Multi-probe K-means (C > 1) to close the gap to brute-force.

HNSW database indexing for better search.

Fine-tuning of WavLM to lift the representation performance.
